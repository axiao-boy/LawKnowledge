# LawKnowledge 技术架构选型与核心考量

> 文档版本：v1.0
> 日期：2026-07-23
> 适用场景：法律知识库（LawKnowledge）静态站点的技术选型与部署架构决策

---

## 一、项目背景与需求

LawKnowledge 是一个面向法律条文、法规解读、案例整理的**知识库类静态站点**。其核心特征：

- **内容型站点**：以文档/条文为主，非交互应用
- **中文为主**：面向国内用户，要求中文排版与搜索体验良好
- **长期维护**：法规会更新，需要版本管理与可持续迭代
- **访问群体在国内**：网络环境是关键约束
- **白嫖优先**：作为知识沉淀项目，倾向于零成本托管

由此推导出的技术要求：

| 要求 | 权重 | 说明 |
|------|------|------|
| 国内访问速度 | ⭐⭐⭐⭐⭐ | 决定可用性 |
| 中文搜索/排版 | ⭐⭐⭐⭐⭐ | 决定阅读体验 |
| 构建产物纯静态 | ⭐⭐⭐⭐ | 便于任意 CDN/OSS 托管 |
| 不依赖国外服务 | ⭐⭐⭐⭐ | 搜索、字体、CDN 等运行时依赖 |
| 可长期维护 | ⭐⭐⭐ | 版本管理、i18n |
| 上手成本 | ⭐⭐⭐ | 个人项目，需快速落地 |

## 三、核心考量因素

选型过程中，围绕"**国内可用性**"这一最高优先级，识别出以下五类核心考量因素：

### 3.1 考量因素一：搜索服务依赖（关键坑）

静态知识库的搜索体验是核心功能，但搜索方案的实现方式直接决定了国内可用性。

| 搜索方案 | 工作方式 | 国内可用性 | 风险 |
|----------|----------|------------|------|
| **本地索引搜索**（minisearch/lunr/Pagefind）| 构建时生成 JSON 索引，浏览器本地查询 | 🟢 无外部依赖 | 无 |
| **Algolia DocSearch** | 调用 Algolia 云端 API | 🔴 国外服务器，慢/超时 | 需申请账号、审批、国内访问慢 |

**结论**：必须选择**本地索引搜索**方案。

- ✅ VitePress（内置 minisearch）、MkDocs Material（lunr）、Astro Starlight（Pagefind）、Hugo
- ❌ Docusaurus 默认 Algolia（需换 `@easyops-cn/docusaurus-search-local` 插件才能本地搜索）

### 3.2 考量因素二：CDN/字体资源依赖

运行时若依赖国外 CDN，国内首屏加载会显著变慢。

| 资源类型 | 风险点 | 处置方案 |
|----------|--------|----------|
| Google Fonts | MkDocs Material 默认引用，国内加载慢 | 禁用 `font: false` 或换国内 CDN |
| jsDelivr CDN | Docsify 默认从该 CDN 加载核心 JS | 换 `cdn.bootcdn.net` / `cdn.staticfile.org` |
| 框架自身资源 | VitePress/Hugo/Starlight 资源自托管 | 无需处理 ✅ |

### 3.3 考量因素三：托管平台（国内可用性的最大变量）

框架选得再好，托管平台不对，国内一样打不开。

| 平台 | 国内速度 | 稳定性 | 站点数限制 | 备注 |
|------|----------|--------|------------|------|
| **GitHub Pages** | 🔴 慢/不稳定 | 差 | 用户主站 1 个/账号；项目站无限 | 免费但国内体验差 |
| **Cloudflare Pages** | 🟢 较好 | 较好 | **无限**（软限 100 项目/账号）| 国内有节点，免费 |
| **Vercel** | 🟡 时快时慢 | 中 | 有限制 | 国内访问不稳定 |
| **Netlify** | 🔴 慢 | 差 | - | 国内访问差 |
| 腾讯云 COS+CDN | 🟢 快 | 好 | 需付费 | 需备案 |
| 阿里云 OSS+CDN | 🟢 快 | 好 | 需付费 | 需备案 |

#### GitHub Pages 的数量限制（常见误区）

- **受限的只有个人/组织主站**（`username.github.io`，每个账号仅 1 个）
- **项目站点其实是无限的**（每个仓库可发布一个 `user.github.io/repo`）

#### Cloudflare Pages 免费版额度

- ✅ 站点数量：**无限**
- ✅ 带宽：**无限**
- ✅ 请求数：**无限**
- ⚠️ 构建次数：500 次/月（每次 git push 触发一次构建）
- ⚠️ 并发构建：1 个
- ⚠️ 自定义域名：每项目 100 个

### 3.4 考量因素四：中文支持与排版

法律知识库内容特点：条文层级深、条款编号、表格化、注解引用。

| 维度 | VitePress | MkDocs Material | Docusaurus |
|------|-----------|-----------------|------------|
| 中文文档完善度 | ⭐⭐⭐⭐⭐ 原生中文 | ⭐⭐⭐⭐ 完整 i18n | ⭐⭐⭐⭐ 25+ 语言 |
| 国内社区活跃度 | 🟢 Vue 团队国内活跃 | 🟢 良好 | 🟡 中等 |
| 多层目录结构 | ✅ 侧边栏分组 | ✅ 最清晰 | ✅ 支持版本 |

### 3.5 考量因素五：长期维护成本

| 维度 | 考量 |
|------|------|
| 构建依赖可获取性 | npm/pip 国内均有镜像，Ruby gem 较慢 |
| 主题可定制性 | React/Vue 框架可定制性 > 纯模板系统 |
| 版本管理需求 | 法规新旧对比场景 → Docusaurus 的 `versioned_docs` 最强 |
| 内容即源 | 所有方案均以 Markdown 为源，可低成本迁移 |

## 四、最终架构选型

### 4.1 推荐主方案

```
┌─────────────────────────────────────────────────────────┐
│  LawKnowledge 技术架构（推荐方案）                        │
├─────────────────────────────────────────────────────────┤
│                                                         │
│  内容层：    Markdown 文档（法律条文、解读、案例）         │
│                  ↓                                      │
│  构建层：    VitePress（Vue + Vite）                      │
│              · 本地搜索（minisearch，无外部依赖）          │
│              · 禁用 Google 字体                          │
│              · 中文原生支持                               │
│                  ↓ git push                              │
│  托管层：    GitHub 仓库（代码版本管理）                   │
│                  ↓ 自动构建                               │
│  发布层：    Cloudflare Pages（构建 + 托管 + CDN）         │
│              · 无限站点/带宽/请求                         │
│              · 国内访问优于 GitHub Pages                  │
│              · 自动 HTTPS / HTTP/3                       │
│                  ↓                                      │
│  访问层：    国内用户通过 Cloudflare 边缘节点访问           │
│                                                         │
└─────────────────────────────────────────────────────────┘
```

### 4.2 选型理由汇总

**为什么选 VitePress？**

1. ✅ **中文生态最完善**：Vue 团队在国内活跃，文档、社区、问题排查均顺畅
2. ✅ **搜索零外部依赖**：内置 minisearch 本地索引，国内访问无延迟
3. ✅ **构建纯静态**：可托管于任意 CDN/OSS，不被平台锁定
4. ✅ **构建速度快**：基于 Vite，大型知识库构建秒级完成
5. ✅ **法律文本排版友好**：Markdown 扩展支持条款编号、表格、注解、目录树

**为什么选 Cloudflare Pages 而非 GitHub Pages？**

1. ✅ **国内访问明显更优**：Cloudflare 在国内有边缘节点
2. ✅ **额度更宽松**：无限带宽/请求/站点（GitHub Pages 有 100GB/月软限）
3. ✅ **站点数量无限**：可同时部署多个知识库子站
4. ✅ **零成本**：免费版完全满足个人知识库需求
5. ⚠️ **唯一限制**：500 次构建/月，需控制 push 频率（攒 commit 再推）

### 4.3 备选方案

| 场景 | 备选方案 | 适用原因 |
|------|----------|----------|
| 需要法规新旧版本对比 | Docusaurus + 本地搜索插件 | `versioned_docs` 版本管理最强 |
| 追求文档结构最清晰 | MkDocs Material + 阿里云 OSS | 侧边栏层级最清晰，需禁用 Google 字体 |
| 10 分钟快速验证 | Docsify | 零构建，浏览器实时渲染 |
| 商业化/高可用 | VitePress + 腾讯云 COS+CDN | 国内最快，需备案 |

## 五、风险与注意事项

### 5.1 已识别风险

| 风险 | 等级 | 影响 | 缓解措施 |
|------|------|------|----------|
| Cloudflare 500 构建/月超限 | 中 | 高频更新时可能无法构建 | 攒 commit 批量推送；或升级 Pro $20/月 |
| Cloudflare 国内访问偶发波动 | 低 | 个别地区/时段变慢 | 可叠加自定义域名 + 国内 CDN |
| 法律内容版权合规 | 高 | 法律法规本身可公开，但解读/案例需注意版权 | 区分"公开法规"与"原创解读" |
| 内容备份 | 中 | 仅依赖 GitHub 单点 | 定期镜像到 Gitee/国内 OSS |
| 搜索中文分词 | 中 | minisearch 默认分词对中文不理想 | 配置中文分词或评估 Pagefind |

### 5.2 实施注意事项

1. **部署配置**：VitePress 需正确设置 `base` 路径（项目站部署到子路径时）
2. **缓存策略**：对 `assets/` 下哈希文件名配置 `Cache-Control: max-age=31536000, immutable`
3. **自定义域名**：Cloudflare Pages 支持免费绑定自定义域名，HTTPS 自动签发
4. **预览部署**：每次 commit 自动生成预览链接，便于内容审阅
5. **搜索索引**：VitePress 本地搜索需确认中文分词效果，必要时切换方案

| 决策项 | 选择 | 核心理由 |

| 搜索方案 | 本地索引（minisearch）| 零外部依赖，国内无延迟 |
| 字体策略 | 禁用 Google Fonts | 避免国内加载慢 |
| 部署触发 | git push 自动构建 | 无需手动操作 |
